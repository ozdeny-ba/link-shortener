# Components — Agent Instructions

Guidelines for all UI component work in this project using shadcn/ui, Radix UI, and Tailwind CSS v4.

> **Rule:** All UI elements must use shadcn/ui components. Never create custom components from scratch when a shadcn/ui primitive exists. Do not install or use any other component library.

---

## Directory Structure

```
components/
  ui/          # shadcn/ui primitives — generated by the CLI, minimal customisation
  <feature>/   # Feature-level components (e.g. components/links/, components/dashboard/)
```

- `components/ui/` is for **primitives only** — buttons, inputs, cards, dialogs, etc.
- Feature-specific or business-logic components belong in a feature folder directly under `components/`.
- Never add business logic (data fetching, auth checks, server actions) inside `components/ui/`.

---

## Adding shadcn/ui Components

Always use the CLI to add new primitives — do not copy-paste from the docs:

```bash
npx shadcn@latest add <component-name>
```

This generates the component into `components/ui/` with the correct imports and variant config. Only modify the generated file to add or adjust **variants** — do not restructure it.

- **Check shadcn/ui first** before building any UI element. If a matching primitive exists (button, input, card, dialog, table, badge, etc.), use it.
- Compose complex UI by combining shadcn/ui primitives — do not recreate what already exists.
- If a component truly has no shadcn/ui equivalent, build it in `components/<feature>/` using Tailwind utilities and document why no primitive applies.

---

## Styling

- Use **Tailwind CSS utility classes** exclusively. Do not write inline styles or external `.css` files (unless adding CSS variables to `globals.css`).
- Compose class strings with `cn()` from `@/lib/utils`:

```tsx
import { cn } from '@/lib/utils';

<div className={cn('base-classes', condition && 'conditional-class', className)} />
```

- Prefer design-token variables (`bg-background`, `text-foreground`, `border-border`, etc.) over hardcoded colours.
- Dark mode is handled through the `dark:` variant — never implement a manual theme toggle unless the design requires it.

## Component Conventions

### Server vs. Client

- Default to **Server Components** — omit `"use client"` unless the component needs:
  - `useState`, `useEffect`, or other React hooks
  - Browser event listeners (`onClick`, `onChange`, etc.)
  - Browser-only APIs (`window`, `document`, etc.)
- When a large component needs a small interactive piece, extract only that piece into a Client Component.

### Props & Types

- Define prop types with `type`, not `interface`:

```ts
type LinkCardProps = {
  slug: string;
  url: string;
  createdAt: Date;
};
```

- Always annotate the return type of exported components: `export default function MyComponent(): React.JSX.Element`.
- Accept a `className?: string` prop on any component that renders a root element, and merge it with `cn()`.

### Icons

Use `lucide-react` for all icons — do not add another icon library:

```tsx
import { Link2, Trash2, Copy } from 'lucide-react';

<Link2 className="size-4" />
```

- Set icon size with `size-*` utilities, not `width`/`height` props.

### Images & Links

- Use `next/image` for all `<img>` elements.
- Use `next/link` for all internal navigation — never use `<a>` for in-app routes.

## Variants with CVA

When building custom components that need multiple visual states, use `class-variance-authority`:

```tsx
import { cva, type VariantProps } from 'class-variance-authority';
import { cn } from '@/lib/utils';

const badgeVariants = cva('inline-flex items-center rounded-full px-2 py-0.5 text-xs font-medium', {
  variants: {
    variant: {
      default: 'bg-primary/10 text-primary',
      destructive: 'bg-destructive/10 text-destructive',
    },
  },
  defaultVariants: { variant: 'default' },
});

type BadgeProps = React.HTMLAttributes<HTMLSpanElement> & VariantProps<typeof badgeVariants>;

export function Badge({ className, variant, ...props }: BadgeProps): React.JSX.Element {
  return <span className={cn(badgeVariants({ variant }), className)} {...props} />;
}
```

## Accessibility

- Always provide meaningful `alt` text for images.
- Use semantic HTML elements (`<button>`, `<nav>`, `<main>`, `<section>`, etc.) before reaching for a generic `<div>`.
- Radix UI primitives handle ARIA attributes automatically — do not override them without a clear reason.
- Ensure interactive elements are keyboard-navigable and have visible focus rings (provided by default via `focus-visible:` utilities in shadcn/ui).
